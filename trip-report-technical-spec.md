# Техническое задание: веб-приложение для ведения Trip-Reports

## 1. Цель проекта

Разработать браузерное веб-приложение для личного ведения trip-reports без системы авторизации. Приложение должно работать как web app с дашбордом, многошаговой анкетой создания и редактирования записей, поиском, архивом, черновиками, историей изменений и хранением данных в БД проекта.[cite:12]

Приложение должно иметь современный, стильный, строгий и минималистичный интерфейс, ориентированный на удобное внесение данных и последующее управление записями. Для длинной анкеты используется многошаговый сценарий заполнения вместо одной длинной формы, что соответствует лучшим практикам проектирования web app и динамических форм.[cite:8][cite:12][cite:15]

## 2. Формат продукта

Тип продукта: browser-based web application для desktop и mobile.[cite:12]

Основные свойства продукта:
- Работа в браузере без установки.[cite:12]
- Без авторизации и ролей доступа.
- Хранение данных в БД проекта.
- Одна основная сущность — trip-report.
- Создание, редактирование, архивирование, удаление, восстановление, поиск и просмотр истории изменений для записей.[cite:10][cite:26][cite:29]

## 3. Основные пользовательские сценарии

### 3.1 Создание новой записи
1. Пользователь открывает приложение и попадает на Dashboard.
2. Нажимает кнопку «Добавить трип-репорт».
3. Проходит многошаговую анкету.
4. Сохраняет запись как черновик или как завершенный trip-report.[cite:28][cite:31]
5. После сохранения запись отображается на Dashboard в соответствующем статусе.

### 3.2 Редактирование записи
1. Пользователь открывает существующую запись из списка.
2. Система открывает ту же многошаговую форму, но с уже заполненными данными.[cite:13]
3. Пользователь изменяет значения на нужных шагах.
4. Сохраняет изменения.
5. Система обновляет запись и добавляет новую запись в историю изменений.[cite:26][cite:29][cite:33]

### 3.3 Сохранение черновика
1. Пользователь заполняет анкету частично.
2. Сохраняет незавершенную запись как draft или система делает автосохранение черновика.[cite:28][cite:34]
3. Позже пользователь открывает черновик и продолжает заполнение с последнего сохраненного шага.[cite:28][cite:31]

### 3.4 Архивирование записи
1. Пользователь выбирает запись из активного списка.
2. Нажимает действие «Архивировать».
3. Запись исчезает из активного списка и попадает в архив.[cite:10][cite:19]

### 3.5 Удаление записи
1. Пользователь выбирает запись.
2. Нажимает действие «Удалить».
3. Система запрашивает подтверждение удаления.[cite:10][cite:16]
4. После подтверждения запись переводится в статус deleted по модели soft delete, а не удаляется физически из БД.[cite:20][cite:24]

### 3.6 Восстановление записи
1. Пользователь открывает архив или список удаленных записей.
2. Выбирает нужную запись.
3. Выполняет восстановление.
4. Система возвращает запись в активный или архивный статус в зависимости от логики реализации.[cite:10][cite:24]

## 4. Основные разделы приложения

### 4.1 Dashboard

Стартовый экран приложения — Dashboard. Он должен отображать ключевые действия и список записей, что соответствует паттерну web app для управления объектами и наборами данных.[cite:10][cite:12]

Состав Dashboard:
- Заголовок страницы.
- Кнопка «Добавить трип-репорт».
- Поле поиска.
- Вкладки или фильтры по статусам: активные, черновики, архив, удаленные.
- Список trip-reports.
- Действия у каждой записи: открыть/редактировать, архивировать, удалить, восстановить (в зависимости от статуса).[cite:10][cite:19][cite:24]

### 4.2 Экран создания записи

Экран создания записи представляет собой многошаговую анкету с индикатором прогресса, навигацией по шагам и валидацией обязательных полей. Такой формат предпочтителен для длинных форм с условной логикой и будущим расширением структуры анкеты.[cite:8][cite:12][cite:27]

### 4.3 Экран редактирования записи

Экран редактирования использует ту же форму, что и создание, но открывается с предзаполненными значениями. Изменение записи должно вести к сохранению новой версии в истории изменений.[cite:13][cite:26][cite:33]

### 4.4 Архив

Архив может быть реализован как отдельная вкладка внутри Dashboard или как отдельный экран. Архивированные записи должны быть доступны для просмотра, поиска, редактирования, восстановления и удаления.[cite:10][cite:19]

### 4.5 Раздел черновиков

Черновики должны быть доступны как отдельный раздел или как фильтр на Dashboard. Пользователь должен видеть незавершенные записи, дату последнего сохранения и иметь возможность продолжить заполнение.[cite:28][cite:31][cite:34]

### 4.6 История изменений

Для каждой записи должен быть доступен просмотр истории изменений. История может быть реализована как отдельный экран, модальное окно, drawer или боковая панель внутри карточки записи.[cite:26][cite:29][cite:33]

## 5. Структура многошаговой анкеты MVP

### Шаг 1. Базовая информация
Поля:
- Дата trip-сессии.
- Вид психоделика: Mushroom / LSD.[cite:12]

Правила:
- Оба поля обязательны.
- После выбора Mushroom пользователь переходит в реализованную ветку Mushroom.
- После выбора LSD система должна поддерживать future extension: в MVP ветка LSD не реализована содержательно, но архитектурно должна быть предусмотрена возможность позднего добавления собственных шагов и полей без переделки формы.[cite:5][cite:12][cite:15]

### Шаг 2. Параметры Mushroom
Поля:
- Дозировка, в граммах.
- Стрейн грибов: Thai Lipa Noi, Tasmanian, Golden Teacher, Cambodian, Mexicana.
- Метод употребления: прямое использование, измельченное использование, lemon tek lemon, lemon tek acid, грибной чай из сухих, грибной чай из подсушенных, грибной чай из сухих грибов, грибной чай из свежих.
- Локация: дома, на природе, event, в гостях.
- Соц-фактор: один, с другом, с девушкой, три и более.
- Время приема, формат HH:MM.

### Шаг 3. Питание перед сессией
Поля:
- Последний прием пищи: сколько часов назад.
- Что кушал.

### Шаг 4. Появление первых эффектов
Поля:
- Через сколько минут появились первые ощутимые эффекты.
- Чем был занят в этот момент: играл, был за компьютером, слушал музыку сидя, слушал музыку лежа, занимался делами, смотрел кино, смотрел YouTube, общался по интернету, общался лично, занятие творчеством, медитация.

### Шаг 5. Заметки / свободный отчет
Поля:
- Заметки / свободный отчет — большое текстовое поле для свободного описания trip-сессии.

## 6. Требования к будущему расширению анкеты

Архитектура формы должна быть изначально рассчитана на постоянное развитие. Новые поля, новые шаги, новые ветки и новые типы психоделиков должны добавляться без полной переработки текущей формы, что соответствует best practice для schema-driven dynamic forms.[cite:5][cite:12][cite:15]

Обязательные требования к расширяемости:
- Форма строится на основе конфигурации шагов и полей.[cite:5][cite:15]
- Каждый шаг описывается схемой: id шага, название, список полей, типы полей, обязательность, порядок, условия показа, валидация.[cite:5][cite:12]
- Поддерживается добавление новых шагов между существующими шагами.[cite:12][cite:15]
- Поддерживается добавление отдельных веток для разных веществ, включая будущую ветку LSD.[cite:5][cite:12]
- Каждая запись хранит версию схемы (`schema_version`), чтобы старые записи оставались корректными после изменения структуры формы.[cite:12][cite:15]

## 7. Валидация формы

Обязательные правила валидации:
- Нельзя перейти на следующий шаг, если обязательные поля текущего шага не заполнены.[cite:8][cite:27]
- Дозировка — только положительное число.
- Время приема — корректное время в формате HH:MM.
- Время появления первых эффектов — только числовое значение в минутах, больше нуля.
- Поля select должны принимать только допустимые значения из конфигурации.

Отображение ошибок:
- Ошибка показывается рядом с конкретным полем, а не только общим сообщением.[cite:8][cite:27]
- Формулировка ошибки должна быть короткой и понятной.
- Валидация должна работать как минимум при переходе между шагами и при финальном сохранении.[cite:27]

## 8. Статусы записей

Система должна поддерживать следующие статусы:
- `draft` — черновик.[cite:28][cite:31]
- `active` — завершенная активная запись.
- `archived` — архивная запись.[cite:10][cite:19]
- `deleted` — запись, удаленная через soft delete.[cite:20][cite:24]

## 9. Черновики

Сохранение черновиков является обязательной функцией. Незавершенная анкета должна сохраняться в БД как полноценная запись со статусом `draft`, а не как временное состояние интерфейса.[cite:28][cite:31]

Требования к черновикам:
- Возможность ручного сохранения черновика.[cite:28]
- Желательно автосохранение при переходе между шагами и/или через заданный интервал времени.[cite:34]
- Сохранение текущего шага анкеты, чтобы пользователь мог продолжить заполнение с нужного места.[cite:28][cite:31]
- Отображение черновиков на Dashboard через отдельный фильтр или раздел.[cite:31][cite:34]

## 10. Архивирование и удаление

Архивирование и удаление — это разные действия и не должны смешиваться.[cite:10]

### Архивирование
- Архивирование убирает запись из активного списка.
- Архивированная запись остается доступной в архиве.
- Архивированную запись можно открыть, просмотреть, редактировать и восстановить.[cite:10][cite:19]

### Удаление
- Удаление работает только через soft delete.[cite:20][cite:24]
- Перед удалением пользователь должен подтвердить действие в модальном окне или confirm-dialog.[cite:10][cite:16]
- Удаленная запись не удаляется физически из БД сразу.
- Удаленные записи могут храниться в отдельном разделе с возможностью восстановления.

## 11. Поиск, фильтрация и сортировка

На Dashboard должен быть реализован поиск по trip-reports. Поиск должен работать как минимум по ключевым структурированным полям и применяться к активным, архивным и черновым записям.[cite:13][cite:19]

Минимальный поиск должен учитывать:
- Дату.
- Вид психоделика.
- Стрейн.
- Локацию.
- Соц-фактор.
- Текст заметок / свободного отчета.

Дополнительные фильтры:
- По статусу: active / draft / archived / deleted.
- По виду психоделика.
- По диапазону дат.

Сортировка:
- По дате trip-сессии.
- По дате создания.
- По дате последнего изменения.
- По умолчанию — новые записи сверху.

## 12. История изменений

Для записей должна вестись история изменений. Audit trail и version history особенно важны для часто редактируемых записей и позволяют отслеживать эволюцию данных во времени.[cite:26][cite:29][cite:33]

Требования:
- При каждом значимом изменении создается запись в истории.[cite:26][cite:29]
- История фиксирует: дату и время изменения, номер версии, тип действия, измененные поля, старое значение, новое значение.[cite:29][cite:33]
- Значимыми изменениями считаются: создание, сохранение черновика, финальное сохранение, редактирование, архивирование, удаление, восстановление.[cite:26][cite:29]
- История изменений должна быть доступна для просмотра из интерфейса записи.[cite:26]

## 13. Требования к интерфейсу

Интерфейс должен быть современным, строгим, минималистичным и визуально чистым, с качественной типографикой, аккуратными карточками, контрастными primary actions и логичным дашбордным сценарием взаимодействия, что соответствует рекомендациям по дизайну web apps.[cite:12]

UX-требования:
- Многошаговая форма вместо одной длинной страницы.[cite:8][cite:27]
- Явный индикатор прогресса: например, «Шаг 2 из 5».[cite:27]
- Кнопки «Назад», «Далее», «Сохранить черновик», «Сохранить trip-report».
- Empty states для пустого списка записей и пустого архива.[cite:12]
- Подтверждение опасных действий, связанных с удалением.[cite:10][cite:16]
- Адаптивность для desktop и mobile.[cite:12]

## 14. Требования к данным и БД

Хранение данных должно быть реализовано в БД проекта. Для поддержки поиска, сортировки и фильтрации часть ключевых полей должна храниться в отдельных колонках, а для будущей гибкости необходимо предусмотреть расширяемую JSON-структуру для дополнительных полей.[cite:11][cite:12]

Рекомендуемая модель хранения:
- Базовая таблица `trip_reports` для ключевых полей.[cite:11]
- JSON-поле `form_payload` или аналогичное поле для дополнительных и будущих данных.[cite:11][cite:12]
- Поле `schema_version` для версии структуры анкеты.[cite:12][cite:15]
- Отдельная таблица `trip_report_history` для истории изменений.[cite:26][cite:33]

## 15. Экспорт данных

Экспорт не является обязательным для MVP, но архитектура приложения должна позволять добавить его позднее без переработки ядра данных.[cite:12]

Приоритеты по экспорту:
- Phase 2: экспорт в JSON.[cite:12]
- Phase 2: экспорт в CSV.
- Phase 3: экспорт в PDF как дополнительный человекочитаемый формат.

## 16. Нефункциональные требования

- Приложение должно корректно работать в современных браузерах на desktop и mobile.[cite:12]
- Интерфейс должен быть адаптивным и удобным на небольших экранах.[cite:12]
- Навигация должна быть понятной и не перегруженной.[cite:8][cite:27]
- Архитектура должна поддерживать рост количества полей, шагов и веток анкеты без переписывания основной логики.[cite:5][cite:12][cite:15]

## 18. Acceptance Criteria

Функционал считается реализованным, если:
- Пользователь может открыть Dashboard и увидеть записи по статусам.[cite:10][cite:19]
- Пользователь может создать новую запись через многошаговую форму.[cite:8][cite:27]
- Пользователь может сохранить запись как черновик и позже продолжить ее заполнение.[cite:28][cite:31]
- Пользователь может завершить trip-report и увидеть его в активном списке.
- Пользователь может открыть запись на редактирование и сохранить изменения.[cite:13]
- Пользователь может архивировать запись и восстановить ее из архива.[cite:10][cite:19]
- Пользователь может удалить запись через soft delete и восстановить ее позже.[cite:20][cite:24]
- Пользователь может пользоваться поиском по записям.[cite:13][cite:19]
- Пользователь может открыть историю изменений записи и увидеть версии и типы изменений.[cite:26][cite:29][cite:33]
- Архитектура формы позволяет в будущем добавить ветку LSD, новые поля и новые шаги без переписывания базового сценария формы.[cite:5][cite:12][cite:15]
